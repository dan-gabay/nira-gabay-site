# Article Candidate Prep

## What this phase does

`scripts/prepare-article-candidate.ts` is Phase 2 of the article import pipeline.

It reads `data/extracted-source-article-preview.json` (output of Phase 1) and produces a structured candidate JSON at `data/article-import-candidate.json` containing everything needed to review and eventually insert a single article into Supabase.

All logic is deterministic, keyword-based, and runs entirely on local files.

---

## What this phase does NOT do

- Does not insert into `public.articles`
- Does not update `article_import_queue`
- Does not publish anything
- Does not delete anything
- Does not call any AI or external API
- Does not rewrite the article body
- Does not invent facts, claims, or therapeutic advice
- Does not use the old source URL as canonical or in any public SEO field

---

## How to run

```bash
# Run Phase 1 first (if not already done)
npm run extract:source-article

# Then run Phase 2
npm run prepare:article-candidate
```

## Candidate JSON structure

| Section | Purpose |
|---|---|
| `source` | Private import metadata (queue ID, original source URL, extraction timestamp) |
| `article_candidate` | The article fields ready for database insertion |
| `seo` | SEO title, meta description, canonical URL, focus keywords, search intent |
| `image_strategy` | New image generation plan (concept, prompt, alt, style, avoid list) |
| `structured_data_candidate` | Article schema.org candidate (not yet injected into app) |
| `internal_linking` | Category, topic cluster, suggested anchors and related queries |
| `migration` | Migration status, redirect needs, old URL as private metadata |
| `quality` | Content length, field completeness checks, warnings list |
| `prepared_at` | ISO timestamp of preparation |

## SEO decisions

**SEO title**
- Appends ` - „†„Щ„®„Ф „Т„С„Р„Щ` if the result fits within ~60 characters.
- Falls back to the clean Hebrew title alone.
- No keyword stuffing. No clickbait.

**Meta description**
- Uses the extracted excerpt as the base.
- If the excerpt is under 140 chars, appends a soft call-to-read.
- Target: 140-165 characters.
- No promises, no generic filler.

**Focus keywords**
- Derived from article title and content via keyword presence checks.
- Reflects what is actually in the article, not invented.

## Slug strategy

- ASCII-only, lowercase, hyphen-separated.
- Generated by matching Hebrew keywords in the title and content text against a priority-ordered concept map.
- Top 2 matching concepts are joined (max 5 words total).
- More specific and primary terms appear first in the map and take priority.
- Hebrew definite articles („Ф) and prefixes („С, „Ю, „Ь) are handled by matching root substrings.
- Fallback: `parenting-article` if nothing matches.

Examples from the map:
- `„Ю„Щ„І„Х„Э` (order/position) вЖТ `birth-order`
- `„Ю„©„§„Ч„™„Щ` (family-related) вЖТ `family-dynamics`
- `„Ц„Х„Т„Щ„Х„™` (couples) вЖТ `couples`
- `„™„І„©„Х„®„™` (communication) вЖТ `communication`

## Canonical strategy

- Canonical URL always points to `https://www.niragabay.com/articles/<slug>`.
- The old source URL is stored only in `source.source_url_private` and `migration.old_source_url_private`.
- The old source URL never appears in: `seo.canonical_url`, `seo.og_title`, `seo.og_description`, `structured_data_candidate`, or any other public field.
- Quality checks enforce this and emit a warning if a violation is detected.

---

## Content cleaning

The following are stripped from `extracted_content` before writing to the candidate:

- The `<h1>` (title stored separately; often contains old-site column prefix)
- The `<h2>` lede (stored separately as excerpt)
- The date/tools `div.clearfix` block
- `<time>` elements
- The first image-only `<p>` (legacy inline article image)
- The author bio block at the bottom (name, credentials, contact email)
- CloudFlare email-protection obfuscated links
- Leading `<br>` from first paragraph
- Empty `<p>` tags

The article body paragraphs and `<strong>` section headings are preserved without modification.

---

## Image generation strategy

- `use_legacy_image: false` - old source images are never used as the final article image.
- `needs_new_image_generation: true` - signals that image generation is required before publishing.
- The concept, prompt, alt, and avoid list are generated deterministically based on the article's theme (family system, couples, parenting, or general self-reflection).
- The prompt targets a modern abstract editorial illustration style: warm amber and ivory tones, organic shapes, premium magazine quality, no literal figures, no text, no logos, no clinical or stock imagery.
- `image_alt` is descriptive Hebrew - not keyword spam.

---

## Internal linking strategy

- `primary_category` and `topic_cluster` are set based on keyword presence in the article.
- `secondary_topics` lists up to 4 related theme areas detected in the content.
- `suggested_anchor_texts` are descriptive Hebrew phrases - no generic "„І„®„Р„Х „Ґ„Х„У".
- `suggested_related_article_queries` are search strings for finding future linking candidates.
- `needs_link_resolution: true` - links are not injected until real article URLs exist.

---

## Structured data candidate strategy

- Uses `Article` schema type.
- `author` and `publisher` are both `{ type: "Person", name: "„†„Щ„®„Ф „Т„С„Р„Щ" }`.
- `main_entity_of_page` is the future canonical URL on `https://www.niragabay.com`.
- `image` is `null` until the new image is generated.
- The old source URL does not appear in the schema.
- Not injected into the app yet. Requires schema validation pass first.

---

## Why this phase is safe

- Reads one local file, writes one local file.
- No network calls. No Supabase reads or writes.
- No AI calls.
- `is_published: false` is hardcoded.
- `migration_status: "candidate_prepared"` makes the stage explicit.
- The quality warnings list surfaces any detected violations before human review.
